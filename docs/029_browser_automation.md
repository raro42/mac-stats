## Install
### DMG (recommended)
[Download latest release](https://github.com/raro42/mac-stats/releases/latest) → drag to Applications.

### Build from source
```bash
git clone https://github.com/raro42/mac-stats.git && cd mac-stats && ./run
```
Or one-liner: `curl -fsSL https://raw.githubusercontent.com/raro42/mac-stats/refs/heads/main/run -o run && chmod +x run && ./run`

### If macOS blocks the app
Gatekeeper may show "damaged" or block the unsigned app—the file is fine. Right-click the DMG → **Open**, then confirm. Or after install: `xattr -rd com.apple.quarantine /Applications/mac-stats.app`

## Tool Agents (what Ollama can invoke)
### Invocation
Whenever Ollama is asked to decide which agent to use (planning step in Discord and scheduler flow), the app sends the **complete list of active agents**: the invocable tools below plus the **SCHEDULER** (informational; Ollama can recommend it for recurring or delayed tasks but cannot invoke it with a tool line). Ollama invokes tools by replying with exactly one line in the form `TOOL_NAME: <argument>`.

## Browser Automation (BROWSER_* tools)
### Requirements
1. **Chromium-based browser** (Google **Chrome**, **Brave**, **Microsoft Edge**, or another Chromium build)
   - **Default macOS:** `/Applications/Google Chrome.app` (standard Chrome install).
   - **Default Linux:** `google-chrome` on `PATH`.
   - **Custom binary:** set **`browserChromiumExecutable`** in `~/.mac-stats/config.json` (or env **`MAC_STATS_BROWSER_CHROMIUM_EXECUTABLE`**) to the absolute path of the browser executable (e.g. Brave or Edge `Contents/MacOS/…`). When this is set, mac-stats **does not** silently fall back to another browser if launch fails.
   - **Optional profile:** **`browserChromiumUserDataDir`** (env **`MAC_STATS_BROWSER_CHROMIUM_USER_DATA_DIR`**) — Chromium `--user-data-dir` for visible launches mac-stats spawns. If the directory does not exist yet, its **parent** must exist so the browser can create it. **Security:** pointing automation at a real daily-driver profile can expose logged-in sessions to BROWSER_* tools and the model; prefer a dedicated profile for agent use if prompts may be untrusted.

2. **Remote debugging on the configured CDP port** (default **9222**)
   - **Port:** **`browserCdpPort`** in `config.json` or **`MAC_STATS_BROWSER_CDP_PORT`** (clamped 1024–65535). Defaults to **9222** if unset.

   - **Operator diagnostics:** From a terminal, run **`mac_stats --browser-doctor`** (same binary as the menu bar app). It prints the **effective** CDP-related settings mac-stats uses (port, timeouts, viewport, Chromium path, `browserToolsEnabled`, optional user-data dir) and probes **`http://127.0.0.1:<port>/json/version`** with the configured HTTP timeout. On success it reports whether **`webSocketDebuggerUrl`** is present and shows **Browser** / **User-Agent** (truncated). On failure it prints short remediation (start Chrome with `--remote-debugging-port`, `chrome://inspect`, first-attach consent). If **`browserToolsEnabled`** is true and nothing answers on the port, the app may also log a **once-per-hour** **info** line under subsystem **`browser/doctor`** in `~/.mac-stats/debug.log` after startup (not every tick).

   - **CDP client timeouts:** **`browserCdpHttpTimeoutSecs`** — per-request HTTP timeout for `GET /json/version` (port checks, retries, and each post-launch readiness probe); default **5** seconds; clamped **1–60**; env **`MAC_STATS_BROWSER_CDP_HTTP_TIMEOUT_SECS`**. **`browserCdpIdleTimeoutSecs`** — max seconds without CDP events before the client drops the WebSocket (visible attach and headless launch); default **600**; clamped **30–3600**; env **`MAC_STATS_BROWSER_CDP_IDLE_TIMEOUT_SECS`**. Legacy **`browserCdpWsConnectTimeoutSecs`** / **`MAC_STATS_BROWSER_CDP_WS_CONNECT_TIMEOUT_SECS`** (clamped **5–120**) is still read when the idle key is unset. With **`-vv`**, mac-stats logs the effective HTTP timeout and CDP idle duration when starting each **new** visible CDP session (subsystem **`browser/cdp`**), and headless launch logs idle at **debug**.

   - **Optional CDP permission grants:** **`browserCdpGrantPermissions`** in `config.json` is a JSON array of Chromium **`Browser.PermissionType`** strings (same names the DevTools Protocol uses: e.g. `geolocation`, `notifications`, `clipboardReadWrite`, `clipboardSanitizedWrite`, `durableStorage`). Default is **empty** (no grants). mac-stats calls **`Browser.grantPermissions`** once when it creates a **new** CDP session (not on every tab switch). Grants are global (no `origin`); failures are logged at **warn** and the session continues. **Microphone/camera-style** types (`audioCapture`, `videoCapture`, …) are never added automatically—include them in the array only if you explicitly want them. Optional env override: **`MAC_STATS_BROWSER_CDP_GRANT_PERMISSIONS`** as a comma-separated list (non-empty replaces the config array).

   - **Optional CDP emulation (viewport + geolocation):** For **CDP paths only** (not HTTP fetch fallback), operators can set **`Emulation.setDeviceMetricsOverride`** and/or **`Emulation.setGeolocationOverride`** via config so responsive layouts and `navigator.geolocation` match a chosen device or region. **Device metrics:** set **both** **`browserCdpEmulateViewportWidth`** and **`browserCdpEmulateViewportHeight`** in `~/.mac-stats/config.json` (integers; both required). Optional **`browserCdpEmulateDeviceScaleFactor`** (default **1.0**) and **`browserCdpEmulateMobile`** (boolean, default **false**). Env: **`MAC_STATS_BROWSER_CDP_EMULATE_VIEWPORT_WIDTH`**, **`MAC_STATS_BROWSER_CDP_EMULATE_VIEWPORT_HEIGHT`**, **`MAC_STATS_BROWSER_CDP_EMULATE_DEVICE_SCALE_FACTOR`**, **`MAC_STATS_BROWSER_CDP_EMULATE_MOBILE`**. **Geolocation:** set **both** **`browserCdpEmulateGeolocationLatitude`** and **`browserCdpEmulateGeolocationLongitude`** (degrees); optional **`browserCdpEmulateGeolocationAccuracy`** (meters). Env: **`MAC_STATS_BROWSER_CDP_EMULATE_GEO_LATITUDE`**, **`MAC_STATS_BROWSER_CDP_EMULATE_GEO_LONGITUDE`**, **`MAC_STATS_BROWSER_CDP_EMULATE_GEO_ACCURACY`**. When metrics or geo keys are **not** fully set, mac-stats calls **`Emulation.clearDeviceMetricsOverride`** / **`clearGeolocationOverride`** on each apply so turning options off in config resets the tab. Emulation is re-applied when mac-stats prepares a CDP tab (before **BROWSER_NAVIGATE**, **BROWSER_SCREENSHOT**, etc.) and affects **all** CDP automation in this process until the cached browser session is replaced (idle timeout or reconnect). **`browser-doctor`** prints the effective emulation summary.

   - **You start the browser:** run your Chromium build with remote debugging so mac-stats can attach, for example:
     ```bash
     /Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --remote-debugging-port=9222
     ```
     Use the same port as **`browserCdpPort`** if you override it. Leave that window running. mac-stats will connect when you use BROWSER_* tools.

   - **mac-stats starts the browser:** if nothing is listening on that port, mac-stats will try to launch the configured (or default) Chromium binary with `--remote-debugging-port=<port>`, then **poll** the debug HTTP endpoint (`/json/version`) until CDP is ready or a deadline passes (default **15s** total wall clock, **250ms** between failed attempts; each HTTP probe uses **`browserCdpHttpTimeoutSecs`**; tune with **`browserCdpPostLaunchMaxWaitSecs`** and **`browserCdpPostLaunchPollIntervalMs`** in `config.json`, or env **`MAC_STATS_BROWSER_CDP_POST_LAUNCH_MAX_WAIT_SECS`** / **`MAC_STATS_BROWSER_CDP_POST_LAUNCH_POLL_MS`**). No manual step needed if the default Chrome install exists on macOS.

3. **If you see CDP / “nothing on port …” or connection errors**
   - Start your browser manually with `--remote-debugging-port=<port>` (matching **`browserCdpPort`**), then retry.
   - If the default Chrome path is wrong, set **`browserChromiumExecutable`** or install/symlink; mac-stats does not install a browser.
   - After a timeout or crash, mac-stats clears the cached session; the next BROWSER_* use will reconnect or relaunch.

### CDP resilience (reconnect, target focus, screenshots)

- **Transport retry:** If a CDP call fails with a WebSocket/connection-style error, the agent clears the cached session and **retries the same tool operation once** (see `browser_agent/mod.rs`, `with_connection_retry`). Logs at **info** include reconnect reason (`websocket_or_transport`, `browser_unresponsive`, or `liveness`) and duration.
- **Health check vs Tokio runtimes:** The tab “`1+1`” CDP health probe (`check_browser_alive`) always runs the evaluate on a plain `std::thread` and waits with `mpsc::recv_timeout` (same budget as before). It does **not** use `Handle::block_on` — nested `block_on` on **current-thread** Tokio runtimes (common for Tauri menu-bar apps) would stall the only executor thread and prevent **all** `tokio::time` deadlines (e.g. heartbeat / agent-router wall-clock) from firing.
- **App shutdown:** mac-stats calls `close_browser_session()` from (1) Tauri [`RunEvent::Exit`](https://docs.rs/tauri/1/tauri/enum.RunEvent.html) (normal quit / window teardown) and (2) a **`ctrlc` handler** for **SIGINT**, **SIGTERM**, and **SIGHUP**, because signal-driven exits often bypass the Tauri event loop before `RunEvent::Exit` runs. The close path clears the cached session, stops CDP trace archival best-effort, and **drops** the CDP client. **Headless** Chrome started by the app gets an optional **SIGTERM** safety net on the known child PID; **visible / user-attached** Chrome (`was_headless == false`) is **never** killed—only the WebSocket closes. Startup still runs `kill_orphaned_browser_processes()` for crash/SIGKILL recovery. At **info**, subsystem **`browser`**: `Browser session closed on shutdown` when a session existed; otherwise a **debug** line notes there was nothing cached. A **debug** line under **`mac_stats::browser_shutdown`** is written when the `ctrlc` handler is registered; on the Tauri path, **`Tauri RunEvent::Exit: closing browser session`** is logged at **debug** before the same close function runs. The debug log file is **flushed and synced** after `close_browser_session()` so lines are visible in `~/.mac-stats/debug.log` even when the process exits immediately. `BROWSER_INTENTIONAL_STOP` is set so in-flight failures during teardown do not trigger the automatic reconnect path.
- **Active tab target:** The CDP target id of the automation tab is remembered after navigate/switch-tab; on later tools, if Chrome’s focused tab drifted (e.g. user clicked another tab), mac-stats tries to **refocus** the stored tab and logs a warning if the stored target no longer exists.
- **Screenshot liveness:** Before capturing a PNG, the agent evaluates `document.readyState` with a **3s** cap. If that fails, the session is cleared and the operation may retry once via the same wrapper (helps with frozen renderers while the WebSocket is still up).
- **Renderer crash (`Target.targetCrashed`):** A second CDP WebSocket calls **`Target.setDiscoverTargets`** with **`{ discover: true }` only** (no `filter` field — current Chrome rejects `filter: null`) and listens for `Target.targetCrashed`. The listener uses a **short TCP read timeout** only so it can poll the session-generation counter while idle; **read timeouts do not tear down the thread** (unlike a mistaken early implementation that exited the loop on every `TimedOut`, so the handler died ~400 ms after attach and never saw `Page.crash`). When the event’s target id matches the **automation tab** (same target id mac-stats tracks after navigate / tab focus), the cached session is cleared, a **warn** line is written under **`browser/cdp`** (truncated target id and URL **host** only), and the next tool entry surfaces `Browser tab renderer crashed; session reset` once so `with_connection_retry` can reconnect—without waiting on health-check timeouts.
- **Smoke test (same session):** From the mac-stats binary, run **`mac_stats --browser-debug-crash-tab -vv`** (after normal config load). That uses **`get_current_tab`** (same bootstrap as BROWSER_* tools), sends CDP **`Page.crash`** on that tab, waits ~1.2s, flushes **`~/.mac-stats/debug.log`**, then exits—so **`Target.targetCrashed`** is handled on the **same** WebSocket endpoint mac-stats uses (no second CDP client). If Chrome on the debug port has **zero** page tabs, empty-browser bootstrap runs **`Browser::new_tab`** on a **worker thread** with an outer **`recv_timeout` of 25s** (`new_tab_with_bootstrap_timeout` in `browser_agent/mod.rs`); stalls beyond that return **Browser unresponsive** and clear the session (headless_chrome’s internal attach wait is **20s**). Prefer a **free** CDP port and a debugging Chrome instance, or open a normal tab manually before automation. For a **long-running** menu-bar process, a **second** `mac_stats` CLI process does **not** share the in-memory session; use this flag **alone** for an automated smoke run, or manually navigate the automation tab to **`chrome://crash`** in the **same** Chrome instance mac-stats attached to.

To disable browser automation entirely (all `BROWSER_*` tools / Chrome-CDP and any HTTP fallback), set `browserToolsEnabled` to `false` in `~/.mac-stats/config.json`. Default is `true`. When disabled, the app refuses any `BROWSER_*` tool call with `Browser tools disabled in config` and does not launch or connect to Chrome; changes apply on the next tool invocation (no restart required).

**CDP navigate URL normalization:** URLs without a scheme still get `https://` prepended for **`BROWSER_NAVIGATE`**. **Browser-internal** targets that the agent already treats as non-network documents (`about:blank` / `about:srcdoc`, including trivial `?` / `#` suffixes, and the same `chrome://new-tab-page` / `chrome://newtab` surfaces as blank-page detection) are **passed through unchanged** and skip HTTP SSRF checks, so CDP **`Page.navigate`** receives a literal `about:blank` (not `https://about:blank`, which Chrome rejects). Internal cleanup paths such as **agent-router turn-timeout** tab reset use the same normalizer.

### Screenshot files (`~/.mac-stats/screenshots/`)

PNG files use a leading `YYYYMMDD_HHMMSS` timestamp in the filename (e.g. `20260321_143000_current.png`). For **BROWSER_SCREENSHOT** with a URL, the host segment of the navigated URL is sanitized for the filesystem (control characters and awkward symbols mapped away; empty or unsafe segments fall back to `page`). After each CDP capture, mac-stats writes the **raw** PNG via a **temp file in the same directory** (`fsync` + `rename`) so a crash mid-write does not leave a truncated PNG as the final name; the **annotated** copy uses the same pattern. Then it writes a second file with the same stem and a **`-annotated`** suffix (e.g. `20260321_143000_current-annotated.png`). The annotated image draws **dashed, type-colored** rectangles and **index labels** (matching the Elements list) using server-side image processing (`getBoundingClientRect` in CSS px, scaled by `window.devicePixelRatio` to screenshot pixels). The **BROWSER_SCREENSHOT** tool result and Discord **`screenshot:saved` / ATTACH** path point at the **annotated** file when annotation succeeds; the raw file remains on disk for debugging. Cross-origin iframe interactables are **not** overlaid (their coordinates are not in the top-level viewport). At **info**, logs include `screenshot annotation wrote …` under **`browser/cdp`**.

On each app startup (after orphaned Chrome cleanup), mac-stats runs **screenshot pruning**: (1) delete files whose timestamp prefix is older than **`screenshotPruneMaxAgeDays`** (default **7**; set to **`0`** to disable age pruning); (2) if total directory size exceeds **`screenshotPruneMaxTotalBytes`** (default **524288000**, i.e. 500 MiB; set to **`0`** to disable the cap), delete **oldest files first** until under the cap. Overrides: env `MAC_STATS_SCREENSHOT_PRUNE_MAX_AGE_DAYS`, `MAC_STATS_SCREENSHOT_PRUNE_MAX_TOTAL_BYTES`. When anything is removed, an **info** line is written to `~/.mac-stats/debug.log` with file count and bytes freed.

On the **agent router** (Discord, scheduler, CLI `run-ollama`, etc.), successful screenshot attachments are also recorded in the assistant reply text as lines `[screenshot: <path>]` so session history and verification retries retain stable references. The CPU-window chat response type includes an **`attachment_paths`** array for the same convention when that path is wired to return files.

### PDF export (`~/.mac-stats/pdfs/`, **BROWSER_SAVE_PDF**)

- **Tool:** `BROWSER_SAVE_PDF: current` (or an empty argument) runs CDP **`Page.printToPDF`** on the **focused automation tab** after a real **BROWSER_NAVIGATE** (same session rules as **BROWSER_SCREENSHOT: current**). Passing a raw URL to **BROWSER_SAVE_PDF** is **rejected** (use **BROWSER_NAVIGATE** first). **`BROWSER_SCREENSHOT: <url>`** is accepted separately: the router forwards it to the same CDP path as internal `take_screenshot`, with the same navigate-time checks as **BROWSER_NAVIGATE**.
- **Files:** Timestamped names `YYYYMMDD_HHMMSS_current.pdf` under **`~/.mac-stats/pdfs/`**, finalized with the same **temp-then-rename** pattern as screenshots. Outbound attachments (e.g. Discord) use the same allowlist as screenshots when the channel supports PDF; the tool result text **always** includes the filesystem path.
- **Size cap:** In-memory PDF bytes are limited by **`browserArtifactMaxBytes`** (same as PNG / other `printToPDF` uses). Oversize exports fail with a clear error (suggest scroll + **BROWSER_SCREENSHOT**, or raising the cap).
- **Print backgrounds:** Default **off** (smaller, text-forward PDFs). Set **`browserPrintPdfBackground`** to `true` in `config.json` or **`MAC_STATS_BROWSER_PRINT_PDF_BACKGROUND=true`** in the environment to include backgrounds.
- **HTTP fallback:** When the agent last succeeded via fetch-based **BROWSER_NAVIGATE** / **BROWSER_CLICK** / **BROWSER_INPUT** fallback (no live Chrome layout), **BROWSER_SAVE_PDF** returns an explicit **not supported** message — there is **no** silent substitute (e.g. no PNG pretending to be a PDF).
- **Logs:** At **info**, subsystem **`browser/cdp`**: `BROWSER_SAVE_PDF wrote …` with byte count and `print_background=`; HTTP-mode refusal logs under **`browser/http_fallback`**.

### Optional CDP trace archive (`~/.mac-stats/traces/`)

For diagnosing CDP hangs, WebSocket issues, or renderer behaviour, operators can enable **Chrome DevTools `Tracing`** for each cached browser session. Default is **off** — when disabled there is **no** extra WebSocket and no `Tracing` traffic.

- **Enable:** `browserCdpTraceEnabled`: `true` in `~/.mac-stats/config.json`, or env **`MAC_STATS_BROWSER_CDP_TRACE_ENABLED=true`**.
- **Wall-clock limit (optional):** `browserCdpTraceWallClockMinutes` (config) or **`MAC_STATS_BROWSER_CDP_TRACE_MINUTES`** — minutes from the **first** `Tracing.start` in this app process; **`0`** = no time limit while enabled.
- **Files:** On session end (browser **idle timeout**, **session clear** after CDP errors, or **app shutdown**), mac-stats calls `Tracing.end`, reads the trace via CDP **`IO.read`** (`ReturnAsStream`, JSON), and writes **`YYYYMMDD_HHMMSS_cdp_trace.json`** under **`~/.mac-stats/traces/`** using the same **temp-then-rename** pattern as screenshots. Open in Chrome **`chrome://tracing`** (Load) or compatible viewers.
- **Limits:** `browserCdpTraceMaxFileBytes` (default **50 MiB**; env **`MAC_STATS_BROWSER_CDP_TRACE_MAX_FILE_BYTES`**), **`browserCdpTraceMaxRetainedFiles`** (default **10** trace files kept; **`0`** disables pruning; env **`MAC_STATS_BROWSER_CDP_TRACE_MAX_RETAINED_FILES`**). Oversize streams are truncated with a **warn** in `debug.log`.
- **Logs:** Subsystem **`browser/cdp`**: `CDP trace: Tracing.start OK …`, `CDP trace: wrote …`, prune lines, and warnings when capture fails or data loss is reported.

### Connection process (step-by-step)

When you invoke a BROWSER_* tool (e.g. BROWSER_NAVIGATE, BROWSER_SCREENSHOT), the app does the following:

1. **Session lookup**  
   mac-stats keeps a single cached CDP (Chrome DevTools Protocol) session. If a session already exists and was used recently (within the idle timeout, default 5 minutes), that connection is reused and the tool runs immediately. **No** post-connect readiness poll runs on reuse.

2. **No session or session expired**  
   The app needs a live Chrome to talk to:
   - **Port check:** It requests `http://127.0.0.1:<browserCdpPort>/json/version` with a per-request timeout from **`browserCdpHttpTimeoutSecs`** (default **5s**) up to a few times with brief pauses between attempts, so a single flaky failure does not immediately trigger a new Chrome launch while Chrome is still starting or right after a restart. The HTTP client uses **`User-Agent: mac-stats/<version>`** (crate version) and **`no_proxy`** so `HTTP_PROXY` / `HTTPS_PROXY` in the environment cannot route loopback CDP discovery through a corporate or dev proxy (which would time out or fail TLS while Chrome is fine). At debug verbosity (`-vv`), the discovery URL (redacted) and User-Agent are logged under `browser/cdp`. If a probe succeeds, it reads `webSocketDebuggerUrl`, **normalizes** it (maps `ws`/`wss` to the right HTTP base for future `/json/*` use, strips DevTools path cruft when deriving bases), **rewrites** bind-all hosts (`0.0.0.0`, `::`) to the discovery host (`127.0.0.1` today) so the WebSocket client can attach, then connects. Connection and discovery errors use **redacted** URL text (no userinfo; sensitive query keys like `token` masked) so diagnostics do not leak credentials. Logs `CDP discovery succeeded on retry` when a later attempt succeeds after an initial miss (visible in `-vv` logs). At debug verbosity, an adjusted-host line is logged when Chromium advertised a bind-all address.
   - **Connect:** It opens a CDP WebSocket with a handshake timeout from **`browserCdpWsConnectTimeoutSecs`** (default **60s**), then **polls** (up to ~15 seconds, ~200 ms between tries) until **`get_tabs`** succeeds reliably. If the tab list is **non-empty**, attach completes immediately. If it is **empty**, mac-stats does **one** extra ~200 ms probe (Chrome sometimes lists zero targets briefly after attach); if the list is still empty, attach **still succeeds** so the agent can create **`about:blank`** on first tool use (see §4b)—a successful handshake alone is still not enough; the client must be able to enumerate targets. Mutex/list failures keep polling until the deadline. At **info** (`-vv`), when attach completes with zero page tabs: `attach readiness OK with 0 page tabs (get_current_tab will bootstrap about:blank if needed)`. If the deadline passes, the connection is dropped and the tool returns a clear error (approve any Chrome debugging prompt, keep Chrome open, retry). Only then is the session cached for subsequent BROWSER_* calls.

3. **Nothing on the configured CDP port**  
   If discovery gives up (no response after the bounded retries):
   - **Visible Chromium (default):** mac-stats starts the configured or default Chromium binary with `--remote-debugging-port=<browserCdpPort>`, optional `--user-data-dir` when **`browserChromiumUserDataDir`** is set, waits about 3 seconds for it to be ready, then connects. The auto-launched command also applies **menu-bar / background automation flags** (e.g. `--disable-renderer-backgrounding`, `--disable-backgrounding-occluded-windows`, `--disable-ipc-flooding-protection`, `--disable-features=CalculateNativeWinOcclusion`, `--disable-blink-features=AutomationControlled`) so painting and CDP stay responsive when the browser is not the foreground window.
   - **Headless:** If the run is configured for headless mode, mac-stats uses the headless_chrome crate to launch a separate process (dynamic debugging port). When **`browserChromiumExecutable`** / **`browserChromiumUserDataDir`** are set, those are passed through to headless launches as well. The same background-rendering flags are merged into that launch (alongside crate defaults and `--disable-software-rasterizer` / `--mute-audio`). That session is still cached the same way.

4. **Session cleared on error**  
   If a CDP call fails with a connection error (e.g. browser closed, network error), mac-stats clears the cached session. The next BROWSER_* use goes back to step 1 and will either reconnect to the configured CDP port or relaunch Chromium (or headless) as needed.

   **mac-stats–spawned visible Chrome:** If mac-stats started visible Chrome on the CDP port (step 3), it tracks that child process. If the OS or user kills that Chrome while the app keeps running, a background wait detects exit and clears the cached session the same way as a transport error—so the next tool use reconnects or launches again. If you attached to Chrome you started yourself (nothing was spawned by mac-stats), there is no child PID to watch; stale handles are still cleared on CDP errors and health checks as before.

4b. **No open tabs after connect**  
   If CDP is connected but Chrome has **zero** page tabs (e.g. every tab was closed), the browser agent creates a single **`about:blank`** tab via CDP, brings it to the front, and continues—so **BROWSER_NAVIGATE**, **BROWSER_CLICK**, and similar tools do not fail with “no tab” until the user can open a tab manually. This does **not** bypass SSRF or navigation policy: only the internal blank page is created automatically; navigations to real URLs still go through the same validation as today. If automatic tab creation fails, the tool returns a clear error (logged at **warn** under `browser/cdp`).

5. **Idle timeout**  
   If no BROWSER_* tool is used for longer than the configured idle timeout (default 5 minutes), the session is dropped. The next use follows steps 2–3 again.

So: **first use** or **after an error/timeout** → check **browserCdpPort** → connect or launch → cache session. **Later uses** → reuse cached session until idle too long or an error occurs.

### Cookie persistence (authenticated sites)

- **File:** `~/.mac-stats/browser_storage_state.json` — JSON snapshot of the Chrome cookie jar from CDP `Network.getAllCookies`, written with an atomic rename (`.json.tmp` → final name).
- **Save:** After each successful **BROWSER_NAVIGATE** (CDP path), and again when the cached session is dropped because of **idle timeout** or **headless vs visible** preference change, so logins survive process restarts and session recycling.
- **Restore:** On each **new** cached CDP session (attach or launch), mac-stats calls `Network.setCookies` once before normal automation, if the file exists. Merging on save uses key `(name, domain, path)`; values from the live browser overwrite the file for matching keys (file-only keys are kept for domains not present in the snapshot).
- **Clear:** **BROWSER_CLEAR_COOKIES** (no argument) deletes the JSON file and calls `Network.clearBrowserCookies`. The file is removed even if CDP is temporarily unavailable (in that case the tool message notes that in-browser clearing could not run).
- **Logs:** Subsystem **`browser/cookies`** at **debug** (`-vv`): counts for save/restore; failures to save/restore are non-fatal and logged at debug.

### Authenticating HTTP(S) proxy (CDP Fetch)

When Chrome’s network stack must authenticate to an **upstream proxy** (407 / proxy `AuthRequired`), automation can otherwise hang until the navigation timeout. If both **`browserCdpProxyUsername`** and **`browserCdpProxyPassword`** are set in **`config.json`** (non-empty strings), mac-stats enables the CDP **`Fetch`** domain with **`handleAuthRequests`** on each automation tab and routes challenges by source:

- **`Proxy`** → supply the configured credentials (same idea as browser-use’s `continueWithAuth` + `ProvideCredentials`).
- **`Server`** (or missing source) → **`Default`** so normal browser handling applies; mac-stats does **not** auto-fill site HTTP basic auth.

Environment overrides: **`MAC_STATS_BROWSER_CDP_PROXY_USERNAME`** and **`MAC_STATS_BROWSER_CDP_PROXY_PASSWORD`**. These values apply only to the **CDP-driven Chrome** session; they are **not** passed to mac-stats’ **`reqwest`** client or **FETCH_URL**. Paused **`Fetch.requestPaused`** requests are still continued by **headless_chrome**’s default interceptor. If **`Fetch.enable`** or listener setup fails, a **warn** is logged under **`browser/cdp`** (no secrets). Navigation timeouts append a short hint mentioning proxy credentials when they may be relevant.

mac-stats vendors a tiny **headless_chrome** patch (`src-tauri/vendor/headless_chrome`) adding **`Tab::reset_fetch_auth_challenge_response_to_default`** so server challenges do not reuse proxy credential state.

### Accessibility tree enrichment and stale index remapping (CDP)

- After the JS interactable pass (same-origin DOM + shadow roots + isolated-world cross-origin iframes), mac-stats runs **DOMSnapshot.captureSnapshot** on the **main document** with whitelisted computed styles (`pointer-events`, `cursor`, `visibility`, `opacity`, `display`), **`includePaintOrder: true`**, and scales layout bounds by **`window.devicePixelRatio`** for CSS-pixel coordinates. The **capture result is parsed as JSON** (not the generated `u32` snapshot structs) so Chrome sentinels like **`-1`** string-table indices do not fail the whole call. It builds a backend-node map to attach **bounding boxes**, drop nodes that are effectively non-hittable from the snapshot’s styles, remove targets **fully covered** by a **higher paint-order** layout box (modal/overlay-style occlusion), and set **`covered=yes`** when a higher layer **intersects** but does not fully contain the box. Failures or older CDPs fall back to the JS-only list (logged at **debug** under **`browser/cdp`**). At **info** (`-v` / default app logging as configured): `DOMSnapshot paint context` and `DOMSnapshot filter` summary lines.
- After collecting interactables, mac-stats calls **Accessibility.getFullAXTree** (main frame and each subframe from **Page.getFrameTree**) and merges **accessible name** and **role** keyed by **backendDOMNodeId**, again from the **raw JSON** `nodes` array so new AX property names (e.g. **`uninteresting`**) do not break the entire merge.
- Each listed element is correlated via **DOM.describeNode** on the same `Runtime` object handle used for clicks, so the LLM snapshot can show lines like `[12] button tag=button role=button bbox=… covered=no "Submit order"` (accessible name when Chrome exposes one; otherwise existing text/placeholder/href fallbacks). When **DOMSnapshot** paint filtering runs, mac-stats **reuses** that `backendNodeId` for the AX merge (one **describeNode** per row instead of two on dense pages); with `-vv`, **debug** may log `interactables AX merge reused backendNodeId for N rows`.
- **BROWSER_CLICK** and **BROWSER_INPUT** keep a copy of the last snapshot’s elements. If the 1-based index no longer points at the same backend node (SPA re-render) or resolution fails, the agent **re-extracts** interactables and remaps by **HTML tag** plus overlapping **identity tokens** (accessible name, visible text, placeholder, `name`, `aria-label`, `href`). A unique match proceeds; zero or multiple matches return a clear error (ambiguous cases list candidate indices). Successful remaps log at **info** under `browser/cdp` (`identity remapped` or `backendNodeId drift`).

### Cookie consent and screenshots
When the user asks to remove or dismiss a cookie consent banner and take a screenshot, the planning prompt instructs the model to include **BROWSER_CLICK** on the consent button (using the Elements list from BROWSER_NAVIGATE) **before** BROWSER_SCREENSHOT. Pre-routing to NAVIGATE + SCREENSHOT is skipped when the question mentions cookie/consent/banner so the planner can add the click step.

### Page load readiness (CDP lifecycle)
- The **headless_chrome** crate enables **`Page.setLifecycleEventsEnabled`** when a tab attaches; mac-stats calls it again on navigation paths so older sessions stay consistent.
- After **`Page.navigate`**, **reload**, and **history** back/forward, mac-stats appends **`Page.lifecycleEvent`** names to a short per-navigation buffer and polls for **`networkIdle`** or **`networkAlmostIdle`** (Chrome may emit either). The poll budget is **4 seconds** when the previous and target URLs are the same **http(s) host** and **10 seconds** when the host differs (invalid or non-http(s) URLs are treated as cross-host for this purpose). If neither event appears in time, the app logs a **warn** under **`browser/cdp`**, inspects **`document.readyState`**, and continues—then runs **`wait_until_navigated()`** against whatever remains of the configured navigation timeout so stuck loads still fail cleanly.
- **Minimum dwell (browser-use `minimum_wait_page_load_time`):** After **`wait_until_navigated`** completes (including the non-timeout SPA/hash fallback path), mac-stats sleeps at least **`browserPostNavigateMinDwellSecs`** (default **1.5**, baseline settle after the load event before SPA readiness and state capture). Operators may lower this for fast static pages or set **0** to skip this sleep. **Same-domain shorter navigation timeout** (`browserSameDomainNavigationTimeoutSecs`) does **not** skip this dwell. Env: **`MAC_STATS_BROWSER_POST_NAV_MIN_DWELL_SECS`**. Clamped **0..=10** seconds. At **`-vv`**, each **`post_nav_stabilization`** min-dwell line includes **`source=`** (`default`, **`env MAC_STATS_BROWSER_POST_NAV_MIN_DWELL_SECS`**, or **`config.json browserPostNavigateMinDwellSecs`**) so a value like **0.25s** is traceable to an override rather than the built-in default.
- **Optional network-quiet wait (browser-use `wait_for_network_idle_page_load_time` style):** When **`browserPostNavigateNetworkIdleEnabled`** is **true**, mac-stats enables **`Network`** on the tab (if not already), tracks in-flight requests via **`requestWillBeSent`** / **`loadingFinished`** / **`loadingFailed`**, and waits until the in-flight set stays empty for **`browserPostNavigateNetworkIdleQuietSecs`** (default **0.5**), capped by **`browserPostNavigateNetworkIdleMaxExtraSecs`** (default **5**; **0** disables this phase while keeping min dwell). Env: **`MAC_STATS_BROWSER_POST_NAV_NETWORK_IDLE_ENABLED`**, **`MAC_STATS_BROWSER_POST_NAV_NETWORK_IDLE_QUIET_SECS`**, **`MAC_STATS_BROWSER_POST_NAV_NETWORK_IDLE_MAX_EXTRA_SECS`**. Default is **off** to preserve latency. The same stabilization runs once per **BROWSER_NAVIGATE** / **new_tab** / **BROWSER_SCREENSHOT** URL nav / back / forward / reload (and after SPA blank-page **reload**). At **`-vv`**, look for **`post_nav_stabilization`** lines under **`browser/cdp`** (min dwell applied, network idle quiet met, or max-extra cap).
- When **`performance.getEntriesByType('resource')`** is available, the **Current page** snapshot can include a line **`Performance resource entries …: N`** so the model sees how many resource timings have been recorded for the navigation so far.

### Navigation timeout, new tab, and go back
- **Navigation timeout:** Maximum wait for BROWSER_NAVIGATE, BROWSER_GO_BACK, BROWSER_GO_FORWARD, and BROWSER_RELOAD is configurable: `config.json` key `browserNavigationTimeoutSecs` (default 30, range 5–120) or env `MAC_STATS_BROWSER_NAVIGATION_TIMEOUT_SECS`. Slow or stuck navigations fail with a clear message (e.g. "Navigation failed: timeout after 30s") plus optional guidance when an authenticating proxy may be involved (see **Authenticating HTTP(S) proxy** above), and a compact `context: ...` line describing CDP discovery/connect and (for BROWSER_NAVIGATE timeouts) whether the tab URL changed (`navchg=0|1`). When **CDP** hits this navigation timeout, the **BROWSER_NAVIGATE** dispatcher **does not** run the HTTP navigate fallback: on the **first** timeout it skips the usual CDP retry + HTTP path (slow loads rarely recover from an immediate second navigate with the same deadline); if the first attempt failed for another reason and the **retry** attempt times out, HTTP fallback is skipped the same way. In both cases **`~/.mac-stats/debug.log`** records an **info** line under **`mac_stats::browser/tools`** (`BROWSER_NAVIGATE: CDP navigation timeout on first attempt…` or `…after retry…`). The tool result stays a real timeout (with `cdp_http=…` / `ws=…` in `context:`) instead of being replaced by a successful HTTP fetch of the same URL.
- **Same-domain shorter timeout (optional):** When the navigation target is on the same domain as the current page (e.g. in-site link or SPA transition), a shorter wait can be used so in-site navigations don’t wait the full timeout. Set `config.json` key `browserSameDomainNavigationTimeoutSecs` (e.g. 5) or env `MAC_STATS_BROWSER_SAME_DOMAIN_NAVIGATION_TIMEOUT_SECS`. When set, same-domain navigations use this value for the post-navigate wait; cross-domain, BROWSER_GO_BACK, BROWSER_GO_FORWARD, and BROWSER_RELOAD still use `browserNavigationTimeoutSecs`. When not set, all navigations use the single navigation timeout. Debug log line "same-domain navigation, using Ns timeout" confirms when the shorter timeout is applied.
- **SPA blank-page retry (default on):** After the usual post-navigate wait, mac-stats measures DOM size and visible text (similar to browser-use). If the page looks like a skeleton (many elements, almost no text) or has no usable body, it waits longer and may `Page.reload` once before giving up. Truly unloadable pages (no `document.body` after retries) return a clear tool error. Toggle with `config.json` `browserSpaRetryEnabled` or env `MAC_STATS_BROWSER_SPA_RETRY_ENABLED`. At verbosity ≥2, look for `SPA readiness` lines in `~/.mac-stats/debug.log`.
- **Wait between browser actions (optional):** To mirror browser-use’s `wait_between_actions`, set `browserWaitBetweenActionsSecs` in `config.json` (fractional seconds allowed) or env `MAC_STATS_BROWSER_WAIT_BETWEEN_ACTIONS_SECS`. After each `BROWSER_*` tool finishes and before the **next** `BROWSER_*` tool in the **same agent request** (including across Ollama follow-up rounds in that request), the app sleeps for that duration if the value is > 0. The **first** browser tool in the request has no leading wait. Values are clamped to 0–60 seconds. Default is `0` (no extra delay). FETCH_URL, RUN_CMD, and other non-browser tools do not trigger or consume this pause. At debug verbosity (`-vv`), look for subsystem `browser/between_actions` (“waiting … between actions”).
- **New tab:** Add `new_tab` after the URL (e.g. `BROWSER_NAVIGATE: https://example.com new_tab`) to open the URL in a new tab and switch focus to it; subsequent BROWSER_CLICK / BROWSER_SCREENSHOT apply to that tab.
- **Screenshot with URL:** `BROWSER_SCREENSHOT: https://…` navigates and captures the **focused** tab (the same internal tab index as `BROWSER_NAVIGATE` and `new_tab`), not the first tab in the Chrome window.
- **BROWSER_GO_BACK:** Use `BROWSER_GO_BACK` (no argument) to go back one step in the current tab's history and get the new page state. Use when returning to the previous page without re-entering the URL.
- **BROWSER_GO_FORWARD:** Use `BROWSER_GO_FORWARD` (no argument) to go forward one step in the **same** tab’s history (after `BROWSER_GO_BACK`). No new URL; operates on the focused tab only.
- **BROWSER_RELOAD:** Use `BROWSER_RELOAD` (no argument) for a normal refresh of the current page, or `BROWSER_RELOAD: nocache` / `BROWSER_RELOAD: hard` for cache-bypass (Shift+reload style). No new URL is supplied—SSRF allowlist is unchanged; the session must already be on a real page.
- **BROWSER_CLEAR_COOKIES:** No argument — removes `browser_storage_state.json` and clears the Chrome cookie jar (see **Cookie persistence** above).
- **Post-navigate tab index reconciliation (OpenClaw-style):** After a successful **BROWSER_NAVIGATE**, **BROWSER_GO_BACK**, **BROWSER_GO_FORWARD**, or **BROWSER_RELOAD**, mac-stats re-enumerates CDP tabs and aligns the internal **focused tab index** with the navigated page’s CDP **`TargetID`** in Chrome’s tab list (with fallbacks: unique URL match, or a single remaining tab). If the previous automation target vanishes from the list and several tabs share the same URL, the agent logs a **warn** and keeps the prior index; if the focused target cannot be recovered, the tool errors with bounded sample URLs. Reconciliation runs **before** building **Current page** state (and cookie-banner refresh) on those paths. At **`-vv`**, look for **`post-navigate tab reconciliation`** under **`browser/cdp`** (`old_index`, `new_index`, `reason=`). **Harness:** from `src-tauri/`, `cargo run --example example_com_history_reload_smoke` drives navigate → navigate → back → forward → reload against live CDP (set `RUST_LOG=mac_stats=debug` to mirror **`-vv`** logging in the example process).
- **Tabs in LLM state:** CDP browser snapshots include a **`Tabs:`** line before **Current page:** listing every open tab with a **0-based** index, URL, title (from CDP target info), and **`(active)`** on the focused tab. This matches **BROWSER_SWITCH_TAB** / **BROWSER_CLOSE_TAB** indices.
- **BROWSER_SWITCH_TAB:** `BROWSER_SWITCH_TAB: <index>` — `Target.activateTarget` / bring to front, update the internal focused-tab index, return full state (Tabs + Elements) for that tab. Only the **first whitespace-separated token** is read as the index (extra text on the same line is ignored); at **info** verbosity, compound args log `using first token only`. Log line: `Browser agent [CDP]: BROWSER_SWITCH_TAB index=…`.
- **BROWSER_CLOSE_TAB:** `BROWSER_CLOSE_TAB: <index>` — closes the tab via CDP (cannot close the **last** remaining tab). Focus moves to the previous tab or tab 0; returns a one-line confirmation plus updated state. Same **first-token** index parsing as **BROWSER_SWITCH_TAB**. Log line: `Browser agent [CDP]: BROWSER_CLOSE_TAB closed index=…`.
- **New tab after click:** If **BROWSER_CLICK** (index or coordinates) increases the tab count (e.g. `target=_blank`), the app focuses the **newest** tab, logs `tab count N -> M`, and prepends a **Note: A new tab opened: …** line to the tool result.
- **Managed tab cap (optional, OpenClaw-style):** `config.json` key `browserMaxPageTabs` or env `MAC_STATS_BROWSER_MAX_PAGE_TABS`. Default **0** = disabled. When set to a positive integer (clamped to **64**), after a successful **BROWSER_NAVIGATE** (including `new_tab`), **BROWSER_CLICK** (index or coordinates), **BROWSER_HOVER**, **BROWSER_DRAG**, or **BROWSER_SCREENSHOT** with a URL, mac-stats closes **other** page tabs in **ascending index order** (oldest in Chrome’s tab strip first) until the open tab count is at most the cap. The **focused** automation tab is never closed; **`CURRENT_TAB_INDEX`** is updated the same way as after **BROWSER_CLOSE_TAB** when tabs are removed. Enumeration or close failures are **fail-open** (the tool still succeeds); at most **one** **warn** per process is logged under **`browser/cdp`** if enforcement is abandoned for a structural reason. At **`-vv`**, look for **`managed tab cap`** lines in `~/.mac-stats/debug.log` (before/after counts and per-close index; no full URLs at **info**). **Harness (bypasses LLM stale-batch):** from `src-tauri/`, `MAC_STATS_BROWSER_MAX_PAGE_TABS=3 cargo run --example managed_tab_cap_smoke` runs one initial navigate plus five `new_tab` navigations sequentially and asserts tab count ≤ cap; use **`RUST_LOG=mac_stats=debug`** to mirror **`managed tab cap`** logging in the example process. If CDP shows **zero** tabs and tab creation times out, open **one** tab manually in the debug Chrome window and re-run (see the example’s module comment).

### Credential placeholders (`BROWSER_INPUT`)
- **File:** `~/.mac-stats/browser-credentials.toml` — if the file is missing on first use, mac-stats creates a commented template and sets permissions to **0600** (Unix).
- **Format:** One TOML table per host pattern. The table key is either an exact hostname (`login.example.org`) or a wildcard (`*.example.org`) matching that host and its subdomains. Inside each table, keys are **placeholder names** and string values are the real secrets.
- **Tool usage:** `BROWSER_INPUT: <index> <text>` may include literals like `<secret>my_password</secret>`. The app replaces each tag with the configured value **after** the model response is parsed, using the **current tab URL’s host** to pick matching tables. Unknown tags or tags with no matching host entry return a clear tool error. Subsystem **`browser/credentials`** logs how many substitutions ran at debug verbosity — **never** the secret values.
- **TOTP:** Placeholder keys whose names end with **`_2fa_code`** must hold a **Base32** shared secret; mac-stats generates a fresh **6-digit SHA1 / 30s** code at substitution time (standard authenticator apps).
- **LLM state:** Password-type inputs no longer expose typed **values** in the **Elements** list (only placeholder text / labels). When any placeholders apply to the current URL, the **Current page** snapshot ends with an **Available credentials** line listing only the `<secret>…</secret>` tags.

### Form-aware `BROWSER_INPUT` (CDP)
- **Metadata:** Each **Elements** row is built with `tag_name`, `input_type`, `contenteditable`, and a **datepicker** heuristic (`datepicker_like`) so routing matches the live control.
- **`<select>`:** Options are chosen by **value** or **visible text** (case-insensitive label match). No per-character key events.
- **HTML5 compound inputs** (`date`, `time`, `datetime-local`, `month`, `week`, `color`, `range`): Uses the **native value setter** from the element prototype, then **`input`** and **`change`** (and **`blur`**) so frameworks observe updates.
- **Datepicker-style text fields:** If `class` contains `datepicker` or `date-picker`, or `data-provide` suggests a datepicker, the same direct-set path is used; optional **`jQuery(el).datepicker('update')`** when present. Matches are logged at **debug** under **`browser/cdp`** (`path=datepicker_heuristic`).
- **`contenteditable`:** Existing text is cleared, then **`document.execCommand('insertText', …)`** (fallback: `textContent`).
- **Default (`text`, `textarea`, etc.):** **`DOM.scrollIntoViewIfNeeded`**, focus via `Runtime.callFunctionOn`, then **`Input.dispatchKeyEvent`** / **`InsertText`** through **headless_chrome** `Tab::type_str` (real keystrokes).
- **Fixture:** `docs/fixtures/browser-input-routing.html` — local file with `<select>`, `<input type="date">`, and a `contenteditable` block for manual checks.
- **`file://` navigation:** **BROWSER_NAVIGATE** (and **BROWSER_SCREENSHOT** with a URL) accept `file:///…` when the path exists on disk and is a regular **`.html`** or **`.htm`** file after canonicalization (symlinks cannot bypass the suffix). The agent’s first-token URL extractor treats `file://…` as a concrete navigation target (same line as `https://…`). Other `file:` types (e.g. `.pdf`) are rejected at pre-navigation. At **debug** (`-vv`), subsystem **`mac_stats::browser/security`** logs `BROWSER_NAVIGATE file:// precheck passed (local html, validated path)` once the path passes those checks (message text intentionally avoids a `[CDP]` prefix so it is not confused with **`browser/cdp`** lines).

### Navigation security (domain allow/block lists)
mac-stats applies a **browser-use–style navigation watchdog** on top of the existing **SSRF** checks (`validate_url_no_ssrf`): only **http** and **https** URLs are subject to domain policy; `about:blank`, `chrome://…`, and similar internal URLs are not filtered this way.

- **Blocklist (always applied):** Defaults ship in the app (`browser_agent/data/default_browser_blocklist.txt`) and include `localhost`, `127.0.0.1`, `::1`, `*.local`, several **cloud console** host patterns, and a short set of **example banking** domains. **Override** the default file entirely by creating **`~/.mac-stats/browser-security-blocklist.txt`** (one hostname or `*.sub.example` glob per line; `#` comments allowed). **Additional** blocked patterns: comma-separated **`BROWSER_BLOCKED_DOMAINS`** or **`MAC_STATS_BROWSER_BLOCKED_DOMAINS`** (environment), the same key in **`~/.mac-stats/.config.env`**, or **`browserBlockedDomains`** in **`config.json`** (string or JSON array).
- **Allowlist (optional):** If **`BROWSER_ALLOWED_DOMAINS`** / **`MAC_STATS_BROWSER_ALLOWED_DOMAINS`**, `.config.env`, or **`browserAllowedDomains`** in **`config.json`** is non-empty, navigation is **only** allowed when the URL host matches at least one entry (exact host, parent domain, or `*.suffix` glob). If unset or empty, any non-blocked host is allowed.
- **When it runs:** Before **BROWSER_NAVIGATE** (CDP), the tool fails with `Navigation to <host> blocked by security policy` if blocked. After CDP navigation settles (including redirects and SPA navigations), the **final document URL** is also checked with the same **SSRF resolver** as the initial target (`validate_url_no_ssrf` / `ssrfAllowedHosts`): if the tab landed on a private or loopback **http(s)** address, the tool **errors** and does not return a normal Elements list. Separately, after navigation and after **BROWSER_GO_BACK**, **BROWSER_GO_FORWARD**, **BROWSER_RELOAD**, **BROWSER_CLICK** / **BROWSER_HOVER** / **BROWSER_DRAG** (when applicable), **BROWSER_SWITCH_TAB**, **BROWSER_CLOSE_TAB**, and **BROWSER_SCREENSHOT** with a URL, the domain allow/block watchdog may reset the tab to **`about:blank`** and prefix a **Security warning** line. Browser-internal schemes (`chrome-error:`, `devtools:`, `chrome:`, etc.) skip the SSRF IP check; `about:blank` / `about:srcdoc` are explicitly allowed; **`file:`** final documents are allowed only when they pass the same local **`.html`/`.htm`** checks as the requested navigation target; **`javascript:`** final documents are rejected. The HTTP fallback path uses **FETCH_URL**-style redirect blocking for **BROWSER_NAVIGATE** and form submissions.
- **Logs:** Subsystem **`browser/security`** at **warn** for pre-navigation blocks and post-navigation resets (`Browser navigation security: …`, `Browser security watchdog: …`).

### Optional LLM screenshot resize (vision verification)
- **Config:** In `~/.mac-stats/config.json`, set both **`browserLlmScreenshotWidth`** and **`browserLlmScreenshotHeight`** (integers, 100–8192). If either key is missing, resize is disabled (full-resolution image bytes are base64-encoded for the vision model, same as before).
- **Scope:** Applies when the first screenshot attachment is sent to the **completion verification** vision model (`commands/verification.rs` → `llm_screenshot`). **Not** applied to PNG files saved under `screenshots/` or to Discord file attachments.
- **Resize:** Decoded with the `image` crate, scaled with **Lanczos3** to the configured size, re-encoded as PNG for Ollama. Logs at INFO: `browser/llm_screenshot: resizing screenshot for LLM from …`.
- **Tests:** `browser_agent` unit tests `scale_click_coords_*` verify linear mapping from LLM image pixels to viewport CSS pixels when resize dims are recorded.
- **Viewport metadata:** Whenever browser state text is built for the model (`format_browser_state_for_llm` / tab-aware snapshots), the app records the **layout viewport** from **`layout_metrics`** (`window.innerWidth` × `innerHeight`) when the CDP preamble is present; otherwise it falls back to **`browserViewportWidth` × `browserViewportHeight`**. Debug under **`browser/llm_screenshot`**: `viewport recorded for LLM coord scaling …`.
- **Layout preamble (CDP):** Before **`Current page:`**, snapshots include **`Viewport: WxH`**, **`Document: WxH`**, and **`Scroll: (x, y)`** from a single `Runtime.evaluate` (layout viewport via `window.inner*`, document size via `documentElement` / `body` scroll dimensions, scroll via `window.scrollX` / `scrollY`). If that evaluate fails, the preamble is omitted (debug log under **`browser/cdp`**) and the rest of the snapshot is unchanged.
- **BROWSER_SCROLL step (CDP):** **`down`** / **`up`** (and the implicit default when the argument is not recognized) use the same layout-metrics evaluate as the preamble: `round(0.9 × viewportHeight)` CSS px, **clamped to 200–2000**. If metrics cannot be read, the step falls back to **500 px** (debug under **`browser/cdp`**). **`top`** / **`bottom`** / explicit pixel values are unchanged.
- **HTTP fallback:** After **`Current page (HTTP):`**, snapshots include **`Source: static HTML parse`** and **`Scroll: (unknown)`** so the model does not treat geometry as live CDP data.
- **JS dialogs (`alert` / `confirm` / `prompt` / `beforeunload`):** CDP registers `Page.javascriptDialogOpening` and auto-accepts or dismisses so automation does not block. After a successful dismiss, the dialog type and a whitespace-normalized, truncated message are appended to a **session-scoped FIFO** (last **3** events, **200** characters per message). Every CDP **Current page** / **Elements** snapshot ends with **`Recent JS dialogs:`** — either **`None`** or one line per retained event (`[alert] …`). The list is cleared when the CDP session is dropped (idle timeout, connection errors, shutdown, or preference-driven relaunch). The HTTP fetch fallback does not observe page JS dialogs; snapshots include **`Recent JS dialogs: (not available in HTTP fallback mode)`** instead.

### BROWSER_CLICK with pixel coordinates (vision flows)
- **Index (default):** `BROWSER_CLICK: <n>` — clicks the *n*th interactable (1-based). The **Elements** list is built from a shared DOM pass (links, buttons, inputs, **select**, **`[contenteditable]`** regions that report `isContentEditable`, common **ARIA roles**, `aria-expanded` / `aria-haspopup`, non-negative **tabindex**, **cursor: pointer** on visible nodes), recursively including **open shadow roots** and **same-origin iframes** (depth limit 3, minimum iframe size 50×50 CSS px). **Cross-origin** iframes are merged via CDP `Page.getFrameTree` + `Page.createIsolatedWorld` + the same JS discovery in each frame (same depth/size limits; total extra rows capped). Listener-based extras still come from **`DOMDebugger.getEventListeners`** on the main document (`DOM.getDocument` uses **`pierce: true`** for that pass). For the **main document**, mac-stats keeps a single retained `Runtime` object for the `__macStatsInteractableNodes()` array: row metadata is produced via **`Runtime.callFunctionOn`** on that array (batch JSON), then each element’s CDP **`objectId`** is taken from `this[i]` on the **same** handle—so list text and handles cannot drift when the DOM mutates between CDP round-trips on heavy SPAs. mac-stats stores those handles after each snapshot so indices stay aligned with **BROWSER_INPUT** and cookie auto-dismiss. The click path runs **`DOM.scrollIntoViewIfNeeded`**, prefers **`DOM.getContentQuads`** (viewport hit), then **`DOM.getBoxModel`** content quad if quads are empty or no usable in-viewport point, then **`getBoundingClientRect`** via `Runtime.callFunctionOn`; then **`Input.dispatchMouseEvent`** via `click_point`, checks **`document.elementFromPoint`** against the target (occlusion → warn + **`element.click()`**), and falls back to **JS `click()`** when geometry or coordinate dispatch fails. Debug lines under **`browser/cdp`** note coordinate vs JS fallback, **`DOM.getBoxModel`** when used, and **`atomic node array + batch metadata`** when the main-world snapshot is built.
- **Coordinates:** `BROWSER_CLICK: coordinate_x=120 coordinate_y=340` (order-independent) or `BROWSER_CLICK: coords 120 340`. Coordinates are interpreted in the same pixel space as the screenshot last sent to a vision model. If LLM resize is enabled and verification used a resized image, the app scales from that image size back to the **recorded** viewport (layout inner size when `layout_metrics` is available, else config `browserViewport*`) before CDP `click_point`. Logs at INFO under `browser/llm_screenshot` when scaling applies. **CDP only** — no HTTP fallback for coordinate clicks.

### BROWSER_HOVER and BROWSER_DRAG (pointer actions, CDP)

- **BROWSER_HOVER:** `BROWSER_HOVER: <n>` — moves the pointer to the same viewport hit point as **BROWSER_CLICK** (shared geometry: `scrollIntoViewIfNeeded`, `getContentQuads`, `getBoxModel` fallback, `getBoundingClientRect`, then `Tab.move_mouse_to_point` → CDP `Input.dispatchMouseEvent` **mouseMoved** only). Warns if the target may be occluded but still moves to the computed point. Counts toward **`MAX_BROWSER_TOOLS_PER_RUN`** like other browser tools. **CDP only** — no HTTP fallback.
- **BROWSER_DRAG:** `BROWSER_DRAG: <from_n> <to_n>` — two 1-based **Elements** indices. Scrolls both nodes into view, resolves viewport centers, then **move** → **mousePressed** (left) → **interpolated mouseMoved** steps (default 8) along the segment → **mouseReleased** at the end. Logs the segment under **`browser/cdp`** at debug. Download detection after the gesture matches **BROWSER_CLICK**. **CDP only** — no HTTP fallback.
- **Focus recovery:** If **`CURRENT_TAB_INDEX`** lands on **`about:blank`**, **`chrome://newtab`**, or **`chrome://new-tab-page`** while at least one other tab has a real URL, **`get_current_tab`** refocuses before pointer tools run: prefer the stored automation **TargetID**, else a tab whose host matches the last automation host, else the first non-ephemeral tab. At **`-vv`**, look for **`focused tab was blank/new-tab; refocused tab index`** under **`browser/cdp`**.
- **Arg parsing:** Router uses the **first line only** of the tool argument for hover/drag indices so a trailing `DONE: success` (or similar) on a second line does not break parsing.

### BROWSER_UPLOAD (file inputs, CDP)
- **Format:** `BROWSER_UPLOAD: <1-based index> <path>`. The index is an **Elements** entry (often the visible “Upload” control); mac-stats resolves the nearest `<input type="file">` by walking up to **five** ancestors and, at each level, checking the node and its **immediate children** only (same idea as browser-use’s “nearby file input”).
- **CDP:** `Runtime.callFunctionOn` on the interactable’s object id (returns the file input’s object id), then `DOM.scrollIntoViewIfNeeded` and **`DOM.setFileInputFiles`** with an **absolute** path. No OS file dialog is opened.
- **Allowed paths (security):** The file must exist, be a non-empty regular file, and its canonical path must lie under **`~/.mac-stats/uploads/`** or **`~/.mac-stats/screenshots/`** only. Paths starting with `~/` are expanded with `$HOME`. Relative paths are resolved against **`uploads/`** (not the process cwd). Other locations are rejected.
- **Logs:** INFO under `browser/cdp` for `BROWSER_UPLOAD index=…` and success.
- **Errors:** If no file input is found near the index, the tool message may list **file-input indices** from the current **Elements** list when available.

### BROWSER_SEARCH_PAGE (text grep, CDP)
- **Format:** `BROWSER_SEARCH_PAGE: <substring>` — case-insensitive literal match over visible text (TreeWalker), up to 20 hits with ~80 chars of context each.
- **DOM path:** Each match includes `[path: body > …]` built from ancestor tags up to `body`.
- **CDP:** `Runtime.evaluate` runs with **`returnByValue: true`** so the `{ matches, total, … }` object is serialized into the protocol `result.value`. Without that, many Chromium builds return only an object handle and `value` is empty — mac-stats would fall back to plain text (no `[path: …]`) or fail scoped search.
- **Scope:** Optional `css_scope=<selector>` or `css_scope:<selector>` at the **end** of the line limits the walk to descendants of the first element matching that selector (e.g. `BROWSER_SEARCH_PAGE: Total css_scope=#cart`).
- **Fallback:** If CDP does not return structured JSON, the agent may fall back to plain `innerText` search **only when** `css_scope` is not set (scoped search requires the JS path).
- **Fused lines:** If the model concatenates another tool into the same argument (e.g. `… css_scope=h1 BROWSER_QUERY: …` or `… DONE: success`), mac-stats truncates at the first embedded known `TOOL:` prefix before parsing so `css_scope` is not polluted.
- **Logs:** Debug under `browser/cdp` includes pattern length and whether a scope was set; debug under `browser/tools` when a fused suffix was stripped.

### BROWSER_QUERY (CSS selector probe, CDP)
- **Format:** `BROWSER_QUERY: <CSS selector>` with optional `attrs=<comma-separated attribute names>` at the end (e.g. `BROWSER_QUERY: nav a attrs=href,class`).
- **CDP:** Same **`returnByValue: true`** evaluate path as **BROWSER_SEARCH_PAGE** so `{ ok, elements, total }` is serialized; otherwise the tool can see “no result from page” when `value` is missing.
- **Result:** Up to **25** elements: tag name, truncated visible text (300 chars), child count, and requested attributes. **`href`** and **`src`** are resolved to absolute URLs via `new URL(..., document.baseURI)`.
- **Purpose:** Cheap structured probe (browser-use–style `find_elements`) before a full **BROWSER_EXTRACT**.
- **Fused lines:** Same truncation rule as **BROWSER_SEARCH_PAGE** — trailing `OTHER_TOOL: …` tokens in the argument are stripped before parsing `attrs=`.
- **Logs:** Debug under `browser/cdp` with selector/attrs summary; debug under `browser/tools` when a fused suffix was stripped.

### Downloads (`~/.mac-stats/browser-downloads/`, CDP)

- **Directory:** `config` helper `browser_downloads_dir()` — same tree as Discord outbound attachments (`extraAttachmentRoots` / screenshot-style allowlist).
- **Setup:** Each focused tab gets `Page.setDownloadBehavior` (allow + path) plus a best-effort auxiliary WebSocket call to `Browser.setDownloadBehavior` (`allowAndName`, `eventsEnabled`) so Chrome names files predictably and emits progress events.
- **After BROWSER_NAVIGATE / BROWSER_CLICK:** The agent waits **~3s**, listens on the auxiliary CDP connection for `Browser.downloadProgress` (`completed`), and diffs the downloads directory against a pre-action snapshot so partial names (`.crdownload`) are ignored. Successful paths are appended to the tool text as `**Download:**` and `[download: <path>]` lines and are added to the router’s `attachment_paths` for Discord when allowed.
- **BROWSER_DOWNLOAD:** Optional timeout in seconds on the tool line (default **30**, max **120**); polls CDP + directory until a new completed file appears or time elapses. With **`-vv`**, INFO lines on target **`browser/cdp`** include **`BROWSER_DOWNLOAD: timeout`** or **`BROWSER_DOWNLOAD: … completed file(s)`** for log review.
- **PDF URLs:** When the focused page URL is **http(s)** and the path ends in **`.pdf`**, mac-stats also saves a **`Page.printToPDF`** copy into the downloads directory (viewer-style PDF tabs).
- **Retention:** On app startup, files in `browser-downloads/` older than **24 hours** (by mtime) are deleted; `debug.log` includes an **`[browser/downloads]`** prefix on the prune line (tracing target is `mac_stats::browser/downloads`; the file formatter omits targets).

### BROWSER_EXTRACT (markdown from live DOM, CDP)
- **Format:** `BROWSER_EXTRACT` (default) or `BROWSER_EXTRACT: no_images` (also **without_images**, **skip_images**, **text_only** — case-insensitive). Default **includes** `![alt](url)` markdown for `<img>` with resolved absolute `src`.
- **Output:** In-page `Runtime.evaluate` walks visible nodes (same visibility idea as interactables), open **shadow roots**, and large same-origin **iframes** (depth/size limits match **BROWSER_SEARCH_PAGE** / interactables). Emits markdown: headings (`#` … `######`), paragraphs, `[text](href)` links, tables (pipe rows + separator after first row), lists, fenced `pre` blocks, optional images. Result is **truncated** to 30k characters, preferring cuts at **`\n\n`** block boundaries when possible.
- **Skeleton hint:** If the DOM has more than 20 elements and normalized `body.innerText` length is less than five times that element count, the tool appends: `Note: this page may still be loading (many elements, very little text).`
- **Fallback:** If JSON from the extractor is missing/invalid, `ok` is false, or markdown is empty after trim, mac-stats logs a warning and uses the previous **innerText**-style pass (`PAGE_TEXT_SHADOW_IFRAME_JS`).
- **Logs:** INFO `BROWSER_EXTRACT: markdown ok …` on success; WARN when falling back to innerText. HTTP-only browser fallback still returns the existing **plain** assembled body text (no markdown path there).

### BROWSER_KEYS (keyboard chords, CDP only)
- **Purpose:** Send **allowlisted** key chords to the **web page** (in-page find, Escape to close dialogs, F5 reload, etc.). This is **not** for Chrome’s own menu bar or UI outside the page.
- **Format:** `BROWSER_KEYS: <chord>` where `<chord>` uses **`+` between tokens with no spaces**, modifiers first: `Meta`, `Ctrl` (or `Control`), `Alt`, `Shift`, then the key.
- **Allowlisted keys:** `Enter`, `Escape` (or `Esc`), `Tab`, `Backspace`, `F5`, or a **single letter `a`–`z`** only when **at least one** modifier is present (e.g. `Meta+f` / `Ctrl+f` for in-page find).
- **Implementation:** CDP `Input.dispatchKeyEvent` via **headless_chrome** (`Tab::press_key_with_modifiers`). **No HTTP fallback** — if Chrome/CDP is unavailable, the tool fails with guidance to use CDP.
- **After dispatch:** Returns the same **Current page** / **Elements** snapshot as other browser tools so the Ollama loop stays grounded.

### Hard navigation failures (Chrome `errorText` and error pages)
CDP **`Page.navigate`** can return an **`errorText`** field (e.g. `net::ERR_NAME_NOT_RESOLVED`, `net::ERR_CERT_AUTHORITY_INVALID`). The **headless_chrome** crate surfaces that as a failed `navigate_to` call. **BROWSER_NAVIGATE** and **BROWSER_SCREENSHOT** (with a URL) turn that into an explicit tool error: the message states that the page did **not** load, includes a **sanitized** Chrome error string (paths such as `/Users/...` are redacted for the model), and tells the model not to treat the tab as the requested site.

When **`errorText`** (or the post-navigate tab state) indicates a **TLS / certificate** problem (`net::ERR_CERT_*`, `ERR_SSL_*`, or similar), the tool result is a **dedicated** paragraph: automation cannot bypass Chrome’s interstitial; the operator may open the same URL once in the **visible** Chrome session attached to CDP and proceed if they accept the risk; **FETCH_URL** is mentioned only as optional **read-only** HTML where host policy allows (it does **not** relax SSRF rules). mac-stats does **not** disable TLS verification in launched Chrome.

If Chrome loads an internal **`chrome-error://`** document without failing `navigate_to`, mac-stats detects that from the tab URL after the navigation wait and returns a similar failure instead of a normal **Elements** list. If **`document.title`** looks like a privacy/TLS interstitial, the failure message uses the same TLS-style guidance. **Cookie-banner** auto-dismiss is skipped on **`chrome-error://`** pages.

When an **HTTP(S)** URL was requested but the tab stays **`about:blank`** and Chrome did not report `errorText`, the tool returns a **single** cautious line that the navigation **may** have failed (heuristic; do not assume the page loaded). This is separate from **SPA / hash** behaviour: if **`wait_until_navigated`** times out with a **timeout** error, the existing **timeout** failure still applies; non-timeout wait failures keep the **warn + short sleep** path when there is no hard error.

Debug logging (`browser/cdp`): **host only** (no path/query) plus an **error class** — for TLS/certificate failures, a short class such as `cert_authority` or `cert_date` at **warn**; otherwise the first token of `errorText` at **debug**. A second **debug** line carries the **full** `errorText` for reviewers.

### LLM snapshot hints (new tab and error pages)
When the focused tab’s URL is a **blank / Chrome new-tab** URL (`about:blank`, `chrome://new-tab-page`, `chrome://newtab`, with the same variants as browser-use), the **Current page / Elements** text sent to the model starts with a short **Note:** line explaining that the tab is not a normal website. When the URL is **`chrome-error://`** or the page **title** suggests a **TLS / privacy** interstitial, the snapshot starts with a one-line **Warning** that numbered elements are not a real loaded site. **BROWSER_EXTRACT** prepends a similar one-line warning in that case. This is agent UX only; SSRF and navigation policy are unchanged.

### Optional page diagnostics (console + uncaught exceptions)
When `config.json` sets `browserIncludeDiagnosticsInState` to `true` (or env `MAC_STATS_BROWSER_INCLUDE_DIAGNOSTICS_IN_STATE=true`), successful **`BROWSER_NAVIGATE`** and **`BROWSER_EXTRACT`** tool results may append a bounded `## Page diagnostics` section (console `error`/`warning` lines and uncaught exception excerpts captured via CDP `Log` / `Runtime`). If there are no diagnostics, the section is omitted. Defaults to **off** to keep tool context size stable. Extract uses a short post-evaluate wait so async `console.*` lines can flush; each navigate/extract run uses fresh buffers (no carry-over from the previous URL).

**No open tabs:** If Chrome exposes CDP but every tab was closed, **`get_current_tab`** creates `about:blank` via **`Target.createTarget`**. The vendored **headless_chrome** `new_tab` wait loop also calls **`register_missing_tabs`** ( **`Target.getTargets`** ) so the page target is attached when **`TargetCreated`** is delayed or missing—reducing **“The event waited for never came”** failures on empty-browser attach.

### Grounded browser retries
- `BROWSER_NAVIGATE` must receive a concrete URL. Natural-language filler such as `BROWSER_NAVIGATE to the video URL` is rejected and treated as an agent-side planning/parsing failure, not as evidence about the website.
- After `BROWSER_NAVIGATE`, `BROWSER_CLICK`, `BROWSER_INPUT`, `BROWSER_UPLOAD`, or `BROWSER_KEYS`, mac-stats caches the latest `Current page` and `Elements` output. Retries should reuse that latest state instead of stale indices from an earlier page.
- If the browser is already on the relevant page and the content is inline, the next retry should inspect that page or click a real listed element. It should not invent a new target URL unless the browser output already exposed one.

### Sequence-terminating navigation
When the model returns multiple tools in one turn, page-changing browser actions (`BROWSER_NAVIGATE`, `BROWSER_GO_BACK`, `BROWSER_GO_FORWARD`, `BROWSER_RELOAD`, `BROWSER_SWITCH_TAB`, `BROWSER_CLOSE_TAB`) **arm the stale-batch guard** for that turn. After one of these actions completes successfully, remaining **index-based** browser tools (`BROWSER_CLICK`, `BROWSER_INPUT`, `BROWSER_UPLOAD`, `BROWSER_KEYS`, `BROWSER_SCROLL`, `BROWSER_EXTRACT`, `BROWSER_SEARCH_PAGE`, `BROWSER_QUERY`, etc.) from the same response are skipped so the model is not acting on stale element lists. **`BROWSER_SCREENSHOT`** and **`BROWSER_SAVE_PDF`** are **not** skipped: they read the **current** page after navigation and are the intended way to batch “navigate then capture” in one model turn. Non-browser tools (`FETCH_URL`, `BRAVE_SEARCH`, `RUN_CMD`, etc.) in the same response still run.

This prevents wrong clicks or inputs caused by stale element indices from the previous page. The model can emit new browser actions in the next turn using the fresh state.

### SSRF protection
All server-side URL fetches and browser navigations triggered by the model are protected against SSRF (Server-Side Request Forgery). Before any HTTP request (`FETCH_URL`) or CDP navigation (`BROWSER_NAVIGATE`, `BROWSER_SCREENSHOT` with a URL), the target URL is validated:

- **Blocked targets:** loopback (127.0.0.0/8, ::1), RFC 1918 private (10/8, 172.16/12, 192.168/16), link-local (169.254.0.0/16, fe80::/10), cloud metadata (169.254.169.254), IPv6 unique-local (fc00::/7), unspecified (0.0.0.0, ::), broadcast, and IPv4-mapped IPv6 variants.
- **Userinfo rejected:** URLs with embedded credentials (e.g. `http://user:pass@host/`) are blocked.
- **DNS resolution check:** The hostname is resolved to IP addresses and each is checked against the blocklist, catching hostnames that resolve to private IPs.
- **Redirect protection:** For HTTP fetches (reqwest), a custom redirect policy checks each redirect hop against the same blocklist, preventing redirect-to-private chains.
- **CDP navigation redirect hops:** For Chrome-driven navigations (`BROWSER_NAVIGATE`, `BROWSER_SCREENSHOT` with a URL, and internal `navigate()` / phone-extract flows), mac-stats enables the CDP **Network** domain and records `Network.requestWillBeSent` events with resource type **Document**. Each distinct URL in the correlated redirect chain (same `loaderId` run, first hop matching the requested URL) is validated with the same `validate_url_no_ssrf` / `ssrfAllowedHosts` rules as **FETCH_URL** and the initial navigation target. **First-hop matching** compares host and path and treats **http** on the default port **80** and **https** on **443** as the same navigation target (so a request typed as `http://…` still correlates when CDP reports `https://…` for the first document load). If hops cannot be correlated (e.g. listener attach failed), enforcement falls back to the existing **post-navigation final URL** SSRF check only; see `browser_agent/mod.rs` (redirect-chain helpers near `cdp_enable_network_for_redirect_chain_capture`).
- **Allowlist:** To explicitly allow fetching from a local service (e.g. a local Redmine), add `"ssrfAllowedHosts": ["hostname-or-ip", ...]` in `~/.mac-stats/config.json`. Default: empty (no exceptions). Each entry matches the **hostname only**: plain values are case-insensitive **exact** match; use `*` / `**` for globs (e.g. `*.app.internal`, `127.0.*` — patterns are anchored to the full hostname so longer names do not accidentally match); use a `contains:` prefix for a deliberate substring rule (e.g. `contains:.staging.`). The same list applies to the initial URL, each redirect hop, and post-navigation final URL checks.
- **HTTP(S) proxy environment variables:** If `HTTP_PROXY`, `HTTPS_PROXY`, or `ALL_PROXY` (or the lowercase variants) are set in the environment for the mac-stats process, **reqwest** (`FETCH_URL`) and **Chrome** (CDP navigations) may route traffic through a proxy. The SSRF pre-check resolves the hostname in Rust **before** the request; that resolution is **not** a complete model of proxy forwarding or re-resolution. For **model-facing** **FETCH_URL** (Discord/router/UI tool loop, HTTP browser fallback) and **BROWSER_NAVIGATE**, mac-stats **logs** once per call (subsystem `browser/security` at info) and **prepends a one-line notice** to the tool result when strict mode is off. **BROWSER_SCREENSHOT** with a URL uses the same strict gate and the same log path when proxies are set but strict mode is off (no text body to prepend). **Background** server-side fetches (e.g. scheduler `FETCH_URL` tasks, generic `fetch_page` command) still honor **strict** fail-closed but do **not** log or prepend, to avoid noise on periodic jobs. Operators who require strict parity between the pre-check and actual egress should **unset** those variables for mac-stats, or set **`strictSsrfRejectWhenProxyEnv`: `true`** in `config.json` (or `MAC_STATS_STRICT_SSRF_REJECT_WHEN_PROXY_ENV=true`) to **refuse** fetches and CDP URL navigations until proxies are cleared or strict mode is turned off again.

### Summary
| Requirement | What mac-stats does |
|------------|----------------------|
| Chromium on CDP port | If the configured port is free, **launches** the configured/default Chromium binary with `--remote-debugging-port=<port>` (and optional `--user-data-dir`). If the port is in use, **connects** to the existing browser. |
| Browser not installed / bad path | Cannot launch until **`browserChromiumExecutable`** points at a valid binary or the default Chrome path exists (macOS); you can always start the browser manually with `--remote-debugging-port=<port>`. |
| Connection dies (timeout, crash) | Session is cleared on error; next use will reconnect to the same port or relaunch. |

### Troubleshooting: Chromium won't start or connect on the CDP port

If BROWSER_* tools fail with CDP discovery errors or launch errors, work through the following.

1. **Binary not at default path**
   - **macOS:** Default is `/Applications/Google Chrome.app/Contents/MacOS/Google Chrome`. For Brave, Edge, or a custom install, set **`browserChromiumExecutable`** to the executable inside the `.app` bundle, or start that browser yourself with `--remote-debugging-port=<browserCdpPort>` and leave it running.
   - **Linux:** Default command is `google-chrome` (must be on `PATH`). Override with **`browserChromiumExecutable`** if needed.

2. **Port already in use**
   - Another process may be using **`browserCdpPort`** (default 9222). mac-stats will *connect* to an existing listener; if that instance is stuck, close it and retry.
   - To see what is using the port (example for 9222): **macOS/Linux:** `lsof -i :9222`. Use your configured port if different.
   - To use a non-default port, set **`browserCdpPort`** (and start the browser with the same `--remote-debugging-port` value).

3. **Spawn fails (e.g. "Launch Chromium …")**
   - **Permissions:** On macOS, ensure the app is allowed to run external apps (e.g. not overly restricted in Privacy & Security). If you built from source, run the binary from Terminal once to rule out Gatekeeper blocking.
   - **Quarantine:** If Chrome was downloaded and is quarantined, the system might block the child process. Try: `xattr -d com.apple.quarantine "/Applications/Google Chrome.app"` (or the path you use).
   - **Architecture:** Use a Chrome build that matches your Mac (Apple Silicon vs Intel). mac-stats does not switch Chrome variants.

4. **Launch succeeds but connection still fails**
   - mac-stats waits ~3 seconds after launch before connecting. On a slow machine or under load, the browser may need longer. Start it manually with `--remote-debugging-port=<port>`, wait until it's fully up, then trigger the BROWSER_* tool again.
   - **Firewall:** Ensure nothing blocks localhost (127.0.0.1) on your **browserCdpPort**. macOS firewall "Block all incoming" can still allow outgoing/local connections; if you use a strict tool, allow the app or browser.

5. **Fallback: headless Chrome**
   - If visible Chrome on 9222 is not an option, the app can use the headless_chrome crate (no port 9222). When the model or user says "headless", BROWSER_* uses that path. You can also retry after a failed launch; in some cases the code falls back to headless automatically.

If the problem persists, check `~/.mac-stats/debug.log` for lines like `Browser agent [CDP]: ...` or `Launch Chrome: ...` to see the exact error.

## Outbound attachments (Discord)

**BROWSER_SCREENSHOT** writes PNGs under `~/.mac-stats/screenshots/`. When the Discord bot (or REST helper) sends those files, it uses the same **central allowlist** as other outbound artifacts: canonical directories under **`screenshots/`**, **`pdfs/`**, and **`browser-downloads/`** inside `~/.mac-stats/` (each applies once that folder exists), plus optional directories from **`extraAttachmentRoots`** in `config.json`. Paths outside that set are skipped so the model cannot attach unrelated files. See **docs/007_discord_agent.md** (Behaviour → outbound file attachments).

- **Artifact size cap:** Raw PNG from CDP, the annotated copy, **BROWSER_SAVE_PDF** output, and other **Page.printToPDF** uses are limited by **`browserArtifactMaxBytes`** in `~/.mac-stats/config.json` (default **10485760**, 10 MiB; clamped between 64 KiB and 100 MiB). If the in-memory capture exceeds the cap, the tool fails with a clear message (no truncated file on disk). Discord and vision verification **stat** attachment paths before reading so oversize files are skipped with a **`browser`** subsystem WARN in `debug.log`.

## Protected config keys (programmatic writes)

Rust code that **writes** `config.json` via merge (e.g. Tauri settings) must not alter security-sensitive top-level keys compared to the current file snapshot. The guard lives in `src-tauri/src/config/protected_mutation.rs` (`reject_if_protected_config_json_changed`): it compares the previous JSON to the would-be next document and refuses the write if any listed key (including nested objects like `toolLoopDetection` or `heartbeat`) would change. This mirrors the OpenClaw “protected paths” idea for automation-driven config mutation. Operators can still change those keys by editing `config.json` directly.
